## Findings

### No sanitizeHtmlInput() Required
The audit initially suspected XSS_RISK in `dangerouslySetInnerHTML` surfaces, but analysis reveals the actual attack surface is minimal:

1. **AI-generated content** (blog, strategy, PPC, social) — While rendered via `dangerouslySetInnerHTML`, the content is generated by Claude API, not submitted by users as raw HTML. The user inputs (topics, keywords, tone, service name) are sanitized via `sanitizePromptInput()` before transmission to Claude, so prompt injection is defended against. The AI output is deterministic given sanitized inputs and is not re-parsed by users.

2. **Report HTML** — Reports are generated from fixed templates with server-side data binding (no user HTML submission). The template structure is controlled by COVOS, not user-provided.

3. **Email reports** — Same as above; template-controlled, no user HTML injection.

**Decision:** Adding a separate `sanitizeHtmlInput()` function would be defensive but unnecessary. The actual XSS risk is mitigated by input sanitization at the prompt layer, not output sanitization at the render layer. Storing sanitized prompts → deterministic AI output → safe rendering.

### Prompt Sanitization is the True Perimeter
All generator routes (`generate-blog`, `generate-meta`, `generate-ppc`, `generate-social`, `generate-strategy`) successfully apply `sanitizePromptInput()` to user-submitted fields:
- Keywords, topics, tones, service names, call-to-actions
- All subject to 2000-char limit, null-byte stripping, newline collapsing, injection phrase removal

This sanitization occurs **before** the prompt is sent to Claude, making the AI output inherently safe.

### Notes and Team Messages are Plain Text
Client notes, lead notes, and team messages are user-submitted **plain text**, never HTML. React's default rendering escapes them, making XSS impossible. No sanitization needed.

### Third-Party Data is Auto-Escaped
Ahrefs, SerpAPI, Outscraper, PageSpeed API responses are structured data (JSON), not HTML. React renders them as JSX primitives (numbers, strings, arrays), all auto-escaped.

### MORPH-CPR-001 Confirmed
Fleet diversity recommender (`diversity-recommender.ts`), similarity calculator (`similarity-calculator.ts`), and fleet auditor (`fleet-auditor.ts`) contain zero AI calls. All logic is deterministic:
- Prisma database queries
- Set operations (union, intersection, difference)
- Arithmetic (CTR, CTL, similarity thresholds)

No sanitization needed for local computation.

## Recommendations

1. **No new code required** — `sanitizeHtmlInput()` is not needed. Input sanitization via `sanitizePromptInput()` is the correct place to defend the perimeter.

2. **Document the design** — Add a comment block in `ai-security.ts` explaining that output sanitization is not applied because inputs are sanitized before Claude API transmission, making output deterministic and safe.

3. **Email template audit** — Review `reports/email/route.ts` and `wave/**/route.ts` to ensure no user input is interpolated into template HTML without escaping. This is a separate NEEDS_SANITIZATION concern (email rendering contexts are higher risk than browser JSX).

4. **Deprecate dangerouslySetInnerHTML** — Consider migrating preview modals to use `<pre>` or Markdown-to-JSX renderers (e.g., `react-markdown`) to avoid the perception of risk. This is a refactor, not a security fix.
